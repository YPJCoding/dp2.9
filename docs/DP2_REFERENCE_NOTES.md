# dp2 参考仓库评估记录

本文档记录 `YPJCoding/dp2` 中可供 `dp2.9` 吸收的经验，以及不建议直接迁移的内容。

评估原则：

- `dp2.9` 继续作为主底板。
- `dp2` 只作为参考仓库，不整包覆盖。
- 能用文档、模板、模块边界吸收的，优先先文档化。
- 涉及运行逻辑的迁移必须单独开关、单独验证。

## 1. README 部署说明

`dp2` 的 README 简洁说明了部署方式：

- 游戏需要已经能正常跑五服并登录。
- 整个 `dp2` 文件夹放在服务端根目录。
- 启动 `df_game_r` 时在命令前加 `LD_PRELOAD=/dp2/libdp2pre.so`。
- 后期一般只编辑和上传 `script/work_reload`，不需要重跑五服。

可吸收项：

- README 中的部署前提可以吸收到 `dp2.9`。
- `LD_PRELOAD` 示例已在 `dp2.9` README 中体现。
- “普通频道加载、PK 频道按需加载”的提醒可以继续保留。

不建议直接照搬：

- `script/work_reload` 的命名大小写与 `dp2` 实际文件 `Work_Reload.lua` 不完全一致，`dp2.9` 后续应统一大小写写法。

## 2. `df_game_r.lua` 热加载机制

`dp2` 的 `df_game_r.lua` 使用 `lfs.attributes` 检查 `/dp2/script/Work_Reload.lua` 修改时间，并通过 `loadfile(filename, "t", new_env)` 加载热更新脚本。

可吸收点：

- 热加载思路有价值。
- `new_env` 只暴露必要上下文，比直接污染全局更好。
- 可用于后续开发环境快速调试 GM/运营脚本。

风险和改进点：

- 热加载脚本一旦写错，可能影响正在运行的服务。
- `new_env` 暴露了 `dp`、`dpx`、`game`、`world`、`logger`、`item_handler`，权限较大。
- 当前实现更适合开发/测试，不建议默认生产启用。

建议迁移方式：

- 不直接迁入主入口。
- 后续创建 `script/hot_reload.lua` 模块。
- 通过 `script/config.lua` 控制是否启用。
- 默认关闭或仅在 debug 环境开启。

## 3. `Work_Reload.lua` 可复用能力

`Work_Reload.lua` 包含大量 GM/运营脚本，主要分为：

- 玩家上下线通知。
- 在线玩家表。
- 全服消息发送。
- 经验副本定时奖励。
- 自动分解/出售装备。
- 翻牌后自动回城。
- 持物进图限制。
- 等级差限制掉落。
- 强化/增幅失败后按道具和概率改结果。
- 聊天指令菜单和 GM 指令。

可吸收项：

- 在线玩家表和上下线 hook，可做成 `script/modules/online.lua`。
- 全服消息函数，可做成 `script/modules/broadcast.lua`，但必须限制频率。
- 自动分解/出售装备逻辑，可作为独立 GM 工具模块。
- 翻牌回城模式可作为可选玩法模块。
- 持物进图、等级差限制掉落可作为玩法规则模块。

高风险项：

- 直接改数据库的聊天指令。
- 发点券、发物品、改等级、改任务、改职业类 GM 指令。
- 强化/增幅失败改成功，会影响装备经济。
- 自动出售/分解容易造成误操作损失。
- 全服广播如果无限制，可能造成刷屏或压力。

建议迁移方式：

- 不整体迁移 `Work_Reload.lua`。
- 拆成独立模块，每个模块配开关。
- GM 指令必须加权限验证。
- 涉及数据修改的命令必须记录日志。
- 默认关闭高风险模块。

## 4. `dp2` 的 `df_game_r.js`

`dp2` 的 `df_game_r.js` 可作为 JS/Frida 功能参考，但不能直接覆盖 `dp2.9`。

可吸收点：

- NativeFunction 的分组思路。
- 部分工具函数和日志写法。
- 某些已验证 Hook 的实现思路。

风险：

- JS 层地址与服务端版本强绑定。
- 不同仓库的 `df_game_r.js` 可能适配不同服务端版本。
- 直接覆盖可能导致入口桥接、Hook 地址、运行时行为混乱。

建议：

- 先完成 `dp2.9` 当前 `df_game_r.js` 的索引和风险标记。
- 再逐项对比 `dp2` 的 JS 函数。
- 每次只迁移一个明确功能，并记录版本、地址、风险和开关。

## 5. `frida.js` 的定位

`frida.js` 不属于 DP 默认加载链路。

处理原则：

- 只能作为外部参考脚本或功能素材。
- 不能默认认为它会被 DP 加载。
- 若要吸收其中逻辑，应迁移到 `df_game_r.js` 或 Lua/DPX 模块中。

## 6. P5 结论

`dp2` 对 `dp2.9` 最有价值的是：

1. README 中的部署说明。
2. `df_game_r.lua` 的热加载设计思路。
3. `Work_Reload.lua` 中可拆分的 GM/运营模块。
4. `df_game_r.js` 中可参考的 Hook/工具写法。

但不建议直接迁移：

1. 整个 `Work_Reload.lua`。
2. 高风险 GM 指令。
3. 未验证的 Hook 地址。
4. 孤立的 `frida.js`。
5. 没有权限控制和日志的发奖/改库逻辑。

最终建议：

- `dp2.9` 保持主底板。
- `dp2` 作为参考素材库。
- 后续吸收内容时按模块、按开关、按风险等级逐项迁移。
